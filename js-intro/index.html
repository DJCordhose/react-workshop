HTML Markup und das DOM

  • HTML Markup wird vom Browser geparst und in das DOM gewandelt
  • Das DOM (Document Object Model) ist eine Baumrepräsentation der dargestellten Objekte
  • Wenn sich das DOM ändert, ändert sich auch die Darstellung
  • Listener auf Änderungen oder Benutzerinteraktionen sind ebenso möglich
  • Das DOM kann den Zustand der Anwendung halten (Eingabe in einem Input-Element)
  • Das DOM kann auch über eine API abgefragt (getElementById) oder verändert (innerHTML = ...) werden
  • Man kann eine komplette Seite nur über die API erstellen

Klassen mit JavaScript

  • Klassen und Konstruktoren sind Mechanismen, um mehrere, strukturell gleiche oder ähnliche Objekte zu erzeugen
  • Auch in JavaScript können eigene Klassen definiert werden
  • Einfachvererbung ist ebenso möglich
  • Grundlage ist die prototypische Vererbung

Prototypen

  • Jedes Objekt hat zusätzlich eine Referenz auf seinen Prototyp
  • Object hat keinen Prototypen, ist aber Prototyp aller anderen Objekte
  • Lesende Property-Zugriffe können transitiv an Prototypen delegiert werden
  • Dies heißt prototypische Vererbung

Setzen des Prototypen aka das Typen-System

Der Prototyp kann nicht direkt, aber durch Aufruf von new gesetzt werden 

/** @constructor */
function Person(name) {
    this.name = name;
}
// Methode
Person.prototype.getName = function() {
    return this.name;
};

var olli = new Person('Olli');
olli.getName() === 'Olli';

Ablauf eines Konstruktoraufrufs mit new

  1. ein leeres, neues Objekt wird erzeugt
  2. die Konstruktor-Funktion hat ein Property prototype, dies wird als Prototyp des neuen Objekts verwendet
  3. this wird an dieses neue Objekte gebunden, d.h. man kann darauf über this zugreifen
  4. die Konstruktor-Funktion wird aufgerufen (mit this gebunden)
  5. das neue Objekt wird implizit zurückgegeben (wenn die Funktion kein explizites return hat)

"Typsystem"

Ein Objekt ist instanceof aller seiner Prototypen 

var olli = new Person('Olli');
olli instanceof Object;
olli instanceof Person;

Module

Module in JavaScript werden über Closures realisiert

Closure in einem Satz

Eine innere Funktion hat immer Zugriff auf alle Variablen und Parameter ihrer äußeren Funktion, auch wenn diese äußere Funktion bereits beendet ist.

Frei nach Douglas Crockford

Beispiel Closure

function outer() {
    var used = "Olli";
    var unused = "Weg";
    return (function() {
        return "Text: " + used;
    });
}

var inner = outer();
console.log(inner());

Closure Definition

Eine Closure ist eine spezielle Art von Objekt, welche zwei Dinge kombiniert

  1. Eine Funktion
  2. die Umgebung in welcher diese Funktion erzeugt wurde - diese Umgebung besteht aus allen lokalen Variablen und Parametern, die sichtbar waren als die Closure erzeugt wurde

Aus der Definition auf MDN

Revealing Module Pattern

var humanresources = (function () {
    function InternalStuff() {
    }

    function Person(name) {
        this.name = name;
        // uses internal stuff
    }

    return {
        Person: Person
    };
})();

Sichtbarkeit bei Revealing Module Pattern

var olli = new humanresources.Person('Olli');
olli.name === 'Olli';
// TypeError: undefined is not a function
new humanresources.InternalStuff();

Geschichte von JavaScript

Anfänge

  • Mai 1995: Brendan Eich, damals Netscape Entwickler, heute Mozilla, stellt Mocha vor
    • September 1995: Der neue Name ist LiveScript
    • Dezember 1995: Sun gibt eine Lizenz auf den Namen Java, Umbenennung auf JavaScript aus Marketinggründen
  • 1996 - 1997: ECMA beginnt mit der Standardisierung für die Umsetzung anderer Browserhersteller
    • Daraus wird ECMAScript 1 als offizieller Standard
    • JavaScript ist eine (sic!) Implementierung

Mittelteil

  • 1998: ECMAScript 2
  • 1999: ECMAScript 3
  • 2000-2009: ECMAScript 4: auf und ab
    • Sehr fortschrittlicher Ansatz
    • u.a. Klassen und deklarierte Typen
    • ActionScript 3 als Implementierung
  • 2005: Ajax und damit SPAs kommen auf
    • Renaissance der Sprache JavaScript
    • Bibliotheken: jQuery, Dojo u.a.
  • 2007: Als Gegenbewegung zu ECMAScript 4 bringen Microsoft und Yahoo ECMAScript 3.1 auf den Weg

Mittelteil #2

  • 2009: ECMAScript 3.1 wird in ECMAScript 5 umbenannt und man spielt wieder miteinander
    • ECMAScript 4 und ActionScript 3 enden als Sackgassen
    • Alle Browser ab IE9, Firefox 3 unterstützen ECMAScript 5
    • ECMAScript 5 hat einen nicht rückwärts kompatiblen strict mode
  • 2011: ECMAScript 5.1 ohne neue Features, lediglich Klärungen in der Spezifikation

Status Quo

  • JavaScript gewinnt immer mehr an Bedeutung
  • Wer für den Browser entwickelt, kommt an JavaScript nicht vorbei
  • JavaScript hatte bisher viele Klippen
    • keine deklarierte Typen
    • Klassen und Vererbung umständlich
    • keine lexikalischen Scopes
    • Bindung von this kann verwirrend sein
    • Kein Einheitliches Modul-Konzept
  • Die aktuelle Version von JavaScript wird diese Klippen entfernen (außer Typen)

JavaScript vs ECMAScript 6 vs ES6 vs ECMAScript 2015

  • ECMAScript ist der Standard und JavaScript ist die Implementierung
  • Ab IE9 sind wir bei ECMAScript 5 (2009)
  • ECMAScript 2015 ist der offizielle, neue Name für ECMAScript 6
  • ES6 ist der kurze Name für ECMAScript 6
  • ES 2015 ist die Abkürzung für ECMAScript 2015
  • ES2016 und ES2017 sind bereits fertig spezifiziert

Gesprengte Klippe #2: Klassen

class Person {
    constructor(name) {
        this._name = name;
    }
    get name() {
        return this._name;
    }
}
class Programmer extends Person {
    constructor(name, language) {
        super(name);
        this.language = language;
    }
    code() {
        return this.name + " codes in " + this.language;
    }
}
const programmer = new Programmer('Erna', 'JavaScript');
console.log(programmer.code());
console.log(programmer instanceof Programmer); // true
console.log(programmer instanceof Person); // true

ES7 aka ES 2016

  • Jedes neue Feature geht durch eine Reihe von Reife-Schritten
  • Jede 'Stage' muss bestimmte Anforderungen erfüllen
    • Stage 0 - Strawman: formloser Vorschlag
    • Stage 1 - Proposal: Komitee untersucht das Feature
    • Stage 2 - Draft: Das Feature wird es wohl machen
    • Stage 3 - Candidate: Implementierungs von Browser-Herstellern für Feedback
    • Stage 4 - Finished: kommt in den Standard (evtl. nicht nächste Version)

Destructuring von Objekten

const person = {
  name: 'Olli',
  address: {
    city: 'Hamburg'
  },
  email: 'oliver.zeigermann@gmail.com'

const {address: {city}} = person;

const person = {

    city: 'Hamburg'

function print({email: contact}) {

// contact=oliver.zeigermann@gmail.com


function g({name: x, y, z=10}) {
  console.log(`x=${x}`); // x=olli

  console.log(`z=${z}`); // z=10

g({ name: 'olli' });

Verbesserungen bei Funktionen

Arrow Functions

const displayInPage = (text) => {
   return document.body.innerHTML +=
       `${text}
`;
};
const displayInPage = text => document.body.innerHTML += `${text}
`;

Default Parameter

const displayInPage = (text = '\n') =>
    console.log(text);
displayInPage();

Rest-Parameters

const displayInPage = (...texts) =>
    texts.forEach(text => console.log(text));

displayInPage('Hi', 'Folks');
// Hi
// Folks
